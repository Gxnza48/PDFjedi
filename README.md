# PDFjedi

> Turn a PDF into a complete digital project: tech spec, UX/UI system, contract, budget, and roadmap — without losing fidelity to what the source document actually says.

PDFjedi is a **modular AI skill system for Claude Code**. It's a curated set of prompts, skills, and templates that work together to convert messy real-world PDFs (briefs, RFPs, proposals, meeting notes, client requirements, design exports) into production-ready artifacts a team can actually execute on.

It is not a single prompt. It is a framework. Skills, templates, and execution prompts compose together so you can run a small piece (e.g. "just extract requirements from this PDF") or the full pipeline ("turn this 40-page RFP into a delivery-ready project").

---

## Why this exists

Reading a PDF and turning it into a *project* is the most under-tooled step in real client work. The information is unstructured, contradictory, partly out-of-date, and almost never maps cleanly onto how an engineering team needs to receive it. Most AI tools produce a generic summary and stop there.

PDFjedi treats that gap as the actual product surface. The output is not a summary — it's the artifacts the team would have built anyway:

- A **technical specification** that an engineer can scope from.
- A **UX/UI system** with screens, components, and interaction notes.
- A **contract** that reflects what was actually agreed.
- A **budget** with line items, assumptions, and risk reserves.
- A **roadmap** with phases, dependencies, and a critical path.

Each artifact is generated by a dedicated skill, against a strict template, and explicitly linked back to the source PDF so you can audit every claim.

## How to use with Claude Code

PDFjedi is consumed as context. You load the relevant files into a Claude Code session and then issue a natural-language instruction. There is no install step.

### Quick start (full pipeline)

In a Claude Code session in this repo, run:

```
Load:
/prompts/master-prompt.md
/skills/*
/templates/*

Then: "Analyze the attached PDF and generate a full project: tech spec, UX/UI system, contract, budget, and roadmap."
```

Attach (or drag in) the source PDF. Claude will follow the master prompt, invoke each skill in order, fill the templates, and emit one artifact per file in an `output/` directory.

### Targeted use (single artifact)

You don't have to run the whole pipeline. Each skill is self-contained.

```
Load:
/prompts/master-prompt.md
/skills/budget-generator.md
/templates/budget-template.md

Then: "Build a budget for the project described in this PDF. Use a fixed-price model with a 15% risk reserve."
```

### Composed use (multi-document)

```
Load:
/prompts/master-prompt.md
/skills/pdf-analysis.md
/skills/tech-spec-builder.md
/templates/tech-spec-template.md

Then: "I'm attaching 3 PDFs (brief, meeting notes, security requirements). Merge them, flag contradictions, then build a tech spec."
```

### What you get back

For the full pipeline, expect a folder like:

```
output/
├── 01-requirements.md           ← normalized, deduped requirements
├── 02-tech-spec.md              ← architecture, modules, APIs, data model
├── 03-ux-ui-system.md           ← screens, components, interactions
├── 04-security-architecture.md  ← threat model + controls
├── 05-roadmap.md                ← phases, milestones, critical path
├── 06-budget.md                 ← line items, totals, assumptions
├── 07-proposal.md               ← client-facing proposal
├── 08-contract.md               ← signature-ready contract draft
└── 99-open-questions.md         ← anything that needs human input
```

Every file links back to the source PDF location (page / section) where each claim came from. If PDFjedi cannot ground a claim, it goes into `99-open-questions.md` instead of being invented.

## Design principles

1. **No invention.** If the PDF doesn't say it, PDFjedi doesn't claim it. Gaps go into `open-questions.md`.
2. **One skill, one job.** Skills are composable. A skill never silently triggers another.
3. **Templates are contracts.** The output schema is fixed so downstream tooling can parse it.
4. **Reasoning is visible.** Every artifact includes a short "How this was derived" footer with citations.
5. **Senior-engineer voice.** Outputs are written the way a senior engineer would write them — direct, decision-oriented, no filler.

## Extending it

Add a new skill by dropping a file in `/skills/` that follows the existing structure (Role, Inputs, Process, Output schema, Quality checks). Add a new artifact type by pairing a skill with a template in `/templates/`. The master prompt auto-discovers everything in those directories.

See [`docs/system-architecture.md`](docs/system-architecture.md) for the full extension model.
