What it does

The most under-tooled step in client work,
turned into a framework.

Reading a PDF and turning it into a project is messy: unstructured, contradictory, partly out-of-date. PDFjedi treats that gap as the actual product surface. The output isn't a summary — it's the artifacts your team would have built anyway.

02 · Tech spec

Technical specification

Architecture, modules, APIs, data model, NFRs — written so an engineer can scope from it without re-reading the PDF.

  • Boring-tech defaults
  • Per-module REQ-ID traceability
  • Round-trip-tested
01 · Requirements

Normalized requirements

Atomic claims, deduped across PDFs, every one cited back to source.

03 · UX/UI

UX/UI system

Screens, components, interaction flows, accessibility rules — written for engineering, not for Figma.

04 · Security

Security architecture

STRIDE threat model, controls, compliance mapping (GDPR, Ley 25.326), launch-readiness checklist.

05 · Roadmap

Roadmap & milestones

Phases, dependencies, parallelism, critical path — calibrated against the budget's PERT estimates.

06 · Budget

Defensible budget

PERT-estimated line items, explicit risk reserve, taxes, payment schedule, optional add-ons, out-of-scope. No invented rates.

07 · Proposal

Client-facing proposal

Plain-language deliverables, roadmap-at-a-glance, investment table, next step.

08 · Contract

Signature-ready contract

Jurisdiction-aware MSA that mirrors the proposal one-to-one. Lawyer review still required.

99 · Open questions

Open-questions register

Every gap, contradiction, and ambiguity surfaced as a question instead of an invention. The register is non-empty by design — a zero count means the model hallucinated.

How to use

Three steps from PDF to output bundle.

There is nothing to install. PDFjedi is loaded as context into Claude Code, then driven with a single natural-language instruction.

  1. 01

    Clone & open

    Bring the repo down locally and open a Claude Code session inside it.

    $ git clone https://github.com/Gxnza48/PDFjedi.git
$ cd PDFjedi
$ claude
  2. 02

    Load the framework

    Tell Claude Code to load the master prompt, every skill, and every template. From here on it is PDFJedi.

    # Load:
/prompts/master-prompt.md
/skills/*
/templates/*
  3. 03

    Attach the PDF & ask

    Drop one or more PDFs into the chat, then issue the instruction. The full bundle lands in output/.

    "Analyze this PDF and generate
 a full project: tech spec, UX/UI
 system, contract, budget, roadmap."

Architecture

Three layers. Plain text. No runtime.

PDFjedi is prompts and contracts, not code. Layers are separated so each can change independently — the philosophy, the process, and the output schema.

L1

Master prompt

/prompts/master-prompt.md

Defines the PDFJedi role, the operating contract (grounding, no-invention, templates-as-contract, reasoning-visible), the pipeline order, and reasoning rules for ambiguous source material. Always loaded first.

L2

Skills (modular)

/skills/*.md

Each skill is one role with strict inputs, a numbered process, an output schema, and quality checks. Skills compose through the master prompt — they never invoke each other.

L3

Templates (structural contracts)

/templates/*.md

Define the exact section structure of every output artifact. Contain no instructions — that's why you can rebrand or relocalize PDFjedi by editing templates alone.
